Symptom

  • When the bot replied with a Markdown table, Slack showed it in a monospace code block that wraps instead of scrolling.
  • Wide rows got visually shattered — separator row broke into multiple dash runs, columns went out of alignment.

Root cause

  • Slack mrkdwn has no real table support.
  • Old fallback: wrap the table in a code block. For anything over ~80 chars per row this is worse than useless.

Fix

  • Detect pipe tables in the replyformatter.
  • Rewrite each body row as a bullet joining header: value fields with —.
  • Slack wraps prose gracefully, so bullets stay readable at any row width.
  • Tables inside fenced code blocks are left alone (those are intentional).
  • Malformed tables left literal (don’t break on bad input).

Decision on the fork

  • Bullets, not file upload. Backlog spec mentioned a file-upload escalation for very wide tables; deferred.
  • File upload would need files:write scope verification, a new attachment path in pb-notify, a 2nd-message error handling path, and a thread-reply-vs-channel decision. Bullets are a strict improvement for every observed case.
  • Commit message documents the deferral. Escalation path stays open if a concrete case forces it.
