Item 4

URL prefix is per-install

When we designed the protocol, there was a temptation to hardcode the prefix everywhere. That would lock the protocol to one specific deploy. Instead, the prefix is now a per-install config value on both sides.

What this buys

The same protocol ships unchanged under different bot names. The default stays the current name, which means zero migration for the running install. A future teammate who wants to run their own bot under a different prefix flips one config on each side + rebuilds; no code change.

Where the value lives

  • Advisor: a new urlSchemeName setting in its persisted config. Read once when the app starts, before registering the protocol with macOS.
  • Slack bot: a new url_scheme_name key in its hook config. Read per call, so a live config update takes effect on the next link emitted.

The edge case we documented

macOS reads the registered URL scheme from the app bundle at install time, not at runtime. So a deploy changing the prefix to something new also has to rebuild the Advisor .app with a matching bundle entry. This is documented in the spec so nobody spins on it later.

Tested

Unit tests on both sides include a “scheme override” case that flips the config to a different prefix via monkeypatching and confirms the URLs flip accordingly. The indirection is real, not cosmetic.
