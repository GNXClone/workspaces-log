URL prefix is per-install

why
Same protocol can ship under other bot names later without code change
default
pittsbot — zero migration for current install
advisor
Settings.urlSchemeName, read once at app-ready
pittsbot
url_scheme_name in state/hook-config.json, read per call

The edge case we documented

  • macOS reads the registered URL scheme from the app bundle at install time, not runtime.
  • So changing the prefix for a new deploy also needs a matching bundle entry + rebuild + reinstall.
  • Documented in the spec so nobody spins on it later.

Tested

  • Both sides have a scheme-override test that monkeypatches the config to a different prefix and asserts URLs flip accordingly.
  • The indirection is real, not cosmetic.
